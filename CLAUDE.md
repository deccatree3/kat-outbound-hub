# KAT Outbound Hub — 캐처스/네뉴 출고 통합 시스템

## 프로젝트 개요

**캐처스(KATCHERS)** 와 **네뉴(네이처뉴트리션 외 4개사)** 의 모든 판매 채널 출고 작업을 통합 자동화하는 도구. 채널마다 흩어진 입력 양식을 표준 출력물(다원 발주서 / EZA 발주서 / KSE Outbound 등)로 변환해 일일 작업 시간을 단축한다.

자매 프로젝트 `kat-kse-3pl-japan` (`C:\claude\kat-kse-3pl-japan`) 은 **일본 KSE 물류비 검토 + 재고 소진 예측** 전용으로 분리 운영. 이 프로젝트는 **출고 일** 전담.

## 사용자/회사 컨텍스트

- 화주(브랜드) 2개:
  - **캐처스**(KATCHERS) — 메인
  - **네뉴**(NENU) = 네이처뉴트리션의 줄임말. + 네이처앤블룸 / 예리코퍼레이션 / 서현커머스 (네이처뉴트리션이 비중 가장 커서 통칭)
- 물류센터:
  - **다원** — 한국 3PL. 캐처스/네뉴 각각 별도 화주 등록.
  - **KSE 일본**(국제익스프레스) — 일본 3PL. Qoo10 일본 출고 전용.
- 운영 도구:
  - **이지오토** — 채널에서 주문 자동수집
  - **이지어드민(EZA)** — 주문/재고/송장 통합 관리. 송장 → 채널 자동 회신.
  - 화주별 EZA↔다원 연동 차이:
    - **네뉴 EZA ↔ 다원**: 자동 연동 ✅ (발주 자동 전달, 송장 자동 등록)
    - **캐처스 EZA ↔ 다원**: 연동 ❌ (수기 발주서, 수기 송장 등록)
- API 제약:
  - 국내몰 API는 보통 1키 = 1용도 (쿠팡 등). EZA가 이미 주요 채널 API 사용 중이라 새 시스템에서 채널 API 직접 호출 어려운 경우 다수.
  - **EZA·다원 WMS는 외부 API 미제공**. 따라서 EZA 다운로드 파일/수기 입력이 우리 시스템의 표준 입력.
  - **Qoo10 일본만 자체 API(QAPI) 사용 가능** — `kat-kse-3pl-japan`에서 검증 완료.

## Qoo10 일본 (Phase 1 이전 대상) — 검증된 사양

자매 프로젝트 `C:\claude\kat-kse-3pl-japan`에서 완성됨. 이전 시 다음을 가져온다:

- **API 클라이언트**: `qoo10/api_client.py`
  - `CertificationAPI.CreateCertificationKey` → SAK 발급
  - `ShippingBasic.GetShippingInfo_v2` (ShippingStat=2 = 신규주문)
  - `ShippingBasic.SetSendingInfo` (송장 등록, OrderNo + ShippingCorp + TrackingNo)
- **출력 빌더**: `qoo10/generator.py`
  - 51컬럼 KSE OMS Outbound xlsx 생성 (`outbound_template.xlsx` 사용)
  - 주소 정제 (CleanSpecialChars VBA 포팅)
- **DB 테이블**:
  - `qoo10_product_mapping` (Qoo10 상품/옵션 → SKU + 수량)
  - `qoo10_pending_brief` (임시 brief 저장)
  - `qoo10_outbound` (출고 이력 + 송장)
  - `qoo10_credentials` (자격증명, 만료일 포함)
- **6단계 stepper UI**: 1.QSM주문가져오기 → 2.출고요청서생성 → 3.KSE등록(수동/안내) → 4.송장가져오기 → 5.QSM송장등록(API/CSV) → 6.안내
- **검증된 사실**:
  - ShippingStat 2 = "On request" = QSM 신규주문
  - ShippingStat 4 = "On delivery" = 배송중 (송장 등록 후 자동 전이)
  - ShippingCorp = "Sagawa" (사가와의 경우)
  - 자격증명 우선순위: env > Streamlit secrets > DB > config.json

## 시작 시점에 결정 필요한 것

1. **DB 정책**:
   - (a) 같은 Supabase 공유 — `kat-kse-3pl-japan`의 `qoo10_*` 테이블 그대로 사용 + 신규 테이블 추가 (`channel_*`, `daone_*` 등) → 빠름, 약간 충돌 위험
   - (b) 새 Supabase 프로젝트 — 데이터 마이그레이션 필요 → 깔끔, 작업 1일
   - (c) 같은 DB + 새 prefix (`hub_*`) — 절충 → 충돌 없음, 마이그레이션 추후

2. **Streamlit Cloud 배포**:
   - 새 GitHub 레포 (예: `kat-outbound-hub`) 생성 후 배포
   - 환경 secrets 별도 등록

3. **Qoo10 일본 코드 이전 방식**:
   - 파일 그대로 복사 후 어댑터 패턴으로 정리
   - DB는 같은 `qoo10_*` 테이블 공유 (새 프로젝트가 운영용으로 자리 잡으면 현재 프로젝트 메뉴 제거)
